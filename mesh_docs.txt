Overview
LLM-Admin is a Template Management System designed to provide prompts optimized for GPT execution, ensuring precise and effective results. LLM-Admin provides a structured approach to template management, ensuring efficiency in GPT-based automation. By leveraging directives, users can finely tune model behavior and enhance template execution capabilities.
Back to Index
Key Features
•	Template support: Users can define prompts tailored for execution.
•	Namespace and Corpora support: A namespace is a dedicated grouping under which multiple corpora templates (vectorized) can be stored.
•	Execution Endpoint: Templates can be executed via a designated OpenAI endpoint.
•	Customization & Reusability: The system allows easy customization and reuse of templates.
•	Caching: Enables faster execution by storing previously computed results, reducing redundant API calls.
•	Asynchronous Execution: Supports running tasks asynchronously, improving efficiency and allowing parallel execution of multiple templates.
•	Streaming Responses: This allows the system to send output progressively, delivering partial results in real time as they are generated. This improves user experience by reducing wait times and enabling faster visibility into long-running operations.

Auto Chunking
 
When this checkbox is enabled, the chunk type, size, and overlap options will be disabled. These values will be automatically assigned by the system based on the uploaded file and will follow the strategy below for auto-chunking the data.
STRATEGY
When inserting documents into a vector database for RAG (Retrieval-Augmented Generation) applications, the chunking strategy significantly impacts retrieval quality. This tool analyzes file content and recommends one of three strategies:
Strategy	Best For	Description
Character	Prose, narratives, reports	Splits by character count with overlap to preserve context
Line	Tabular data, CSV, JSON-lines	Splits by line boundaries to preserve record integrity
Automatic	PowerPoint files	Delegates to external slide-based processing pipeline
Usage
Standard Mode
Analyzes the file and returns a uniform chunking recommendation
Output:
{"strategy":"character","size":800,"overlap":100}
Boundary Detection Mode
Analyzes the full file, detects structural boundaries (for documents) or intelligent groupings (for tabular data), and returns variable-sized chunks
Outputs:
Output (variable chunks detected):
{
"strategy": "character",
"chunks": [
{"size": 295, "overlap": 35},
{"size": 388, "overlap": 46},
{"size": 214, "overlap": 25}
],
"boundary": true
}
Output (boundary detection not optimal, falls back to uniform):
{"strategy":"line","size":5,"overlap":0,"boundary":false}
JSON Lines Generation
Converts tabular data (CSV/TSV with headers) to JSON Lines format. The generatedFile field contains the path to the created .jsonl file.
Output:
{
  "strategy": "line",
  "size": 60,
  "overlap": 0,
  "generatedFile": "/path/to/data.jsonl"
}
Combining Options
The --boundaries and --generateJSONLines were combined to prduce the output as:
{
"strategy": "line",
"size": 60,
"overlap": 0,

File Type Detection
Files are categorized by extension:
Extension	Type	Processing
.pdf	PDF	PDFBox text extraction
.doc, .docx	Word	Apache POI extraction
.xls, .xlsx	Excel	Apache POI cell-by-cell extraction
.ppt, .pptx	PowerPoint	Returns automatic immediately
.txt, .csv, .json, .jsonl, .xml, .md, .log, .yaml, .yml	Text	Direct file read
Other	Text	Treated as plain text (best effort)
Text Extraction
Each file type has a specialized extractor:
•	PDF: Uses Apache PDFBox PDFTextStripper to extract all text content
•	Word (.docx): Uses Apache POI XWPFWordExtractor for Office Open XML format
•	Word (.doc): Uses Apache POI WordExtractor for legacy format
•	Excel: Iterates through all sheets, rows, and cells; joins cell values with tabs, rows with newlines
•	Text files: Direct UTF-8 file read
Content Analysis
The analyzer samples content and computes the following metrics:
Metric	Description	Used For
Average Line Length	Mean character count per non-empty line	Detecting prose vs. tabular
Line Length Coefficient of Variation	Standard deviation / mean	Measuring consistency
Delimiter Ratio	(commas + tabs + pipes) / total characters	Detecting structured data
JSON Pattern Detection	Regex matching for [{...}] or {...} per line	Identifying JSON-lines
CSV Structure Detection	Consistent delimiter count across lines	Identifying tabular data
Chunking Strategies
Character-Based Chunking
When Used: Prose, narratives, reports, documentation, and free-form text.
How It Works: Content is split into chunks of N characters with M characters of overlap.
Document: "The quick brown fox jumps over the lazy dog. The dog was sleeping."
Chunk 1 (chars 0-30):   "The quick brown fox jumps over"
Chunk 2 (chars 20-50):  "jumps over the lazy dog. The d"  ← 10 char overlap
Chunk 3 (chars 40-66):  "dog. The dog was sleeping."     ← 10 char overlap
Character Chunk Sizes
Size	Overlap	When Recommended	Rationale
500	75	Technical documentation	Smaller chunks for precise retrieval
800	100	Standard prose, reports	Balanced paragraph-level context
1200	150	Long-form content, academic papers	Preserves broader context
Line-Based Chunking
When Used: Tabular data (CSV/TSV), JSON-lines, structured records, log files.
How It Works: Content is split by line boundaries with no overlap.
CSV Data:
id,name,email
1,John,john@example.com
2,Jane,jane@example.com
With size=2 (2 lines per chunk):
Chunk 1: "1,John,john@example.com\n2,Jane,jane@example.com"
Automatic (PowerPoint)
When Used: PowerPoint files (.ppt, .pptx).
Returns immediately without content analysis:
{"strategy":"automatic","size":0,"overlap":0}
Detected Boundary Patterns
For character-based content (documents):
Pattern	Example
Markdown Headers	## Introduction
Numbered Sections	3.2 Methodology
Chapter/Section Labels	Chapter 5
ALL CAPS Headers	EXECUTIVE SUMMARY
Horizontal Rules	-------------------
Double Blank Lines	Paragraph separators
Page Breaks	Form feed or [Page X] markers
HTML/XML Tags	<section>, <h1>-<h6>
For line-based content (tabular):
•	Blank lines (record group separators)
•	JSON array boundaries
•	Comment lines (starting with #)
Boundary Optimality
The analyzer automatically evaluates if boundary detection provides value. It falls back to uniform chunking (boundary: false) when:
Condition	Reason
No internal boundaries detected	Only start/end found
Too few chunks (< 3)	No benefit from variable sizing
Too many chunks (> 100)	Over-segmentation
All chunk sizes identical	Uniform chunking is equivalent
Chunk sizes too uniform (CV < 0.15)	No meaningful variation
Degenerate chunking	One chunk has >70% of content
JSON-lines format	Uniform chunking is better

Intelligent Line Chunking
When --boundaries is used with CSV or JSON-lines files, the analyzer performs intelligent grouping based on common values in leading columns/properties.
How Intelligent Chunking Works
1.	Inspect leading columns: Examines the first 1-3 columns of each row
2.	Detect groupings: Identifies consecutive rows with the same value
3.	Calculate chunk sizes: Returns chunk sizes matching natural data groups
4.	Optimize: If all groups are the same size, returns uniform chunking

Chunk Type:
Character - Character chunking is a method of splitting text into smaller segments based on a fixed number of characters.
Chunk character size - Chunk character size defines the maximum number of characters included in a single chunk when a file is split for processing (for example, during embedding or indexing). It determines how large each chunk of text will be.
Larger chunk sizes contain more context but may:
Increase processing time
Reduce precision in retrieval
Smaller chunk sizes:
Improve search accuracy
May lose contextual continuity
- Default max characters per chunk: 1200
- Default overlap: 200
- If the chunk size is less than 500, a warning pop-up will appear on save.
Chunk character overlap - Chunk character overlap defines how many characters are repeated between consecutive chunks.
Overlap helps preserve context between chunks, especially when sentences or ideas span chunk boundaries.
The overlapping characters from the end of one chunk are repeated at the beginning of the next chunk.
Line - Line chunking splits text based on line boundaries instead of character count.
Each line—or a group of lines—is treated as a chunk.
Chunk line size
Chunk line size defines the maximum number of lines included in a single chunk when a file is split for processing (for example, during embedding or indexing). It determines how many lines of text are grouped together as one chunk.
Chunk line overlap
Chunk line overlap defines how many lines are repeated between consecutive chunks.
Overlap helps preserve context between chunks, especially when meaningful content spans multiple lines.
The overlapping lines from the end of one chunk are repeated at the beginning of the next chunk, ensuring continuity during processing and retrieval.
- If the chunk count is greater than 100, a warning pop-up will appear on save.
- Default lines per chunk: 25
- Default overlap: 3
•	Behavior When Changing Chunk Type
o	When switching from character → line or line → character, the system will reset the chunk configurations and display a confirmation pop-up before applying the change.

Namespaces
Overview
Namespaces serve as logical containers that group multiple templates under a common vector type. Each namespace is tied to the vectorization method selected—ensuring that all templates within it follow the same vector storage and processing rules. This structure helps maintain consistency, improve organization, and enables efficient management of vectorized content across the system.
A namespace is a dedicated grouping under which multiple templates can be stored and vectorized. When a vector type is selected (Faiss, Qdrant, or BM25), the corresponding namespace is created specifically under that vector type. All templates added to this namespace will inherit the same vectorization method.
We can assign a template to a particular namespace by using the namespace dropdown on the create corpora pop-up.
 
•	When creating a new Corpora template, you can use the Create New option in the namespace dropdown.
•	This option allows you to generate a new namespace on the fly without leaving the template creation flow.
 
Generate Manifest
This option creates manifest templates for all the associated templates under the namespace. The Manifest template will include a human-readable summary, key topics, document type, and example questions that the file could answer. This metadata will improve discoverability, search filtering, governance, and debugging. It will also map the file to the chunks/vectors generated from it.
Example:
 
The manifest templates will be created with the template name followed by _manifest.
 
The manifest templates will be placed in a new namespace that is suffixed with _manifest, followed by the namespace name.
EDIT NAMESPACE
When we edit an existing namespace,
•	We cannot change the vector type
•	We cannot delete a namespace when there are associated templates
•	When we edit the name of the namespace, the respective name should be updated on all the associated manifest templates.
•	When we delete a template under the namespace, the corpora template will be deleted and the respective chunks will be deleted from the Qdrant storage as well.
•	When we update the content of a corpora template, the chunking process starts all over and the status will be changed to Queued.
Creating a Namespace Directly from the Namespace Page
•	Navigate to the Namespaces page.
•	Click the Create button.
•	A new, empty namespace will be created.
•	Templates can later be added to this namespace as needed.
•	Every namespace that we create will be automatically added to the namespace dropdown while creating the Corpora template.
 
Using the ‘Default’ Namespace Option During Corpora Template Creation
•	When creating a Corpora template, selecting the Default option in the namespace dropdown will automatically create a namespace using the current template name as the namespace name.
•	This is useful for quickly generating a one-to-one mapping between a template and its corresponding namespace.

In both the UI and SVC endpoints, prompts support directives.
Directives will customize the behavior and parameters of backend service calls to the LLM. They are defined at the model level but can be overridden within a template.
Directive Format:

Directive Types:
•	Model Directives
•	Custom Directives

Model Directives
These parameters are passed to the LLM and define behavior:
GPT Model Directives
1.	max_tokens: Sets the maximum tokens in a response.
•	Specifies the maximum number of tokens to generate in the response. Tokens can be as short as one character or as long as one word.
Example: [[max_tokens=150]]
2.	temperature (temp): Controls the randomness of the response. Lower values make the output more deterministic and focused, while higher values increase variability and creativity.
Example: [[temp=0.7]] or [[temperature=0.7]]
3.	response_format: Defines the format in which the response is generated. This can be plain text, JSON, or other formats depending on the API's capabilities.
•	Note : Need to surround with double quote around value for response_format.
Example:
•	[[response_format="{\"type\": \"text\"}"]]
•	[[response_format="{\"type\": \"json_object\"}"]]
•	[[response_format={"type":"json_schema","json_schema":{"name":"math_response","strict":true,"schema":{"type":"object","properties":{"steps":{"type":"array","items":{"type":"object","properties":{"explanation":{"type":"string"},"output":{"type":"string"}},"required":["explanation","output"],"additionalProperties":false}}},"required":["steps"],"additionalProperties":false}}}]]
4.	model: Specifies the model to use for generating responses. Different models have varying capabilities, sizes, and performance characteristics. This overrides even the model specified at runtime or the model defined in the JSON itself. The model directive provided in the template prompt takes the highest precedence above all.
Example: [[model=gpt-4o-mini]]

Custom Directives
Custom directives are internal commands used to modify templates or retrieve external data. At runtime, each directive is replaced with the corresponding output generated by its associated action. These directives are defined and managed within our system. They are primarily used for internal operations such as updating template content or fetching data from a URL to populate sections of the template dynamically .
Types of Custom Directives
URL_GET
Fetches data from a URL via GET request.
Example: [[URL_GET=http://example.com/fetchdata]]
EMBED_TEMPLATE
Includes one template within another; execution fails if the template is missing.
Syntax: [[EMBED_TEMPLATE="{"template":"template_name","fallback" : "fallback_template_name_1, fallback_template_name_3"}"]]
Example: [[EMBED_TEMPLATE="{"template":"{{Variable_1}}","fallback" : "{{Variable_2}}, {{Variable_3}}"}"]]
Note: Variable_1, Variable_2, and Variable_3 are parameter names that serve as template names provided during template testing.
EMBED_TEMPLATE_NOFAIL:
Embeds a template without failing if the template is missing.
Syntax: [[EMBED_TEMPLATE_NOFAIL="{"template":"template_name","fallback" : "fallback_template_name_1, fallback_template_name_3"}"]]
Example: [[EMBED_TEMPLATE_NOFAIL="{"template":"{{Variable_1}}","fallback" : "{{Variable_2}}, {{Variable_3}}"}"]]
Note: Variable_1, Variable_2, and Variable_3 are parameter names that serve as template names provided during template testing.

EMBED_TEMPLATE_VECTOR
This is similar to an embed template, but this takes the vectorized value of the mentioned template for processing.
Syntax: [[EMBED_TEMPLATE_VECTOR="{"template":"template_name","prompt":"user_prompt","resultcount":{{Count}},"similarity_threshold":{{Thershold_value}}, "fallback":"fallback_template_name"}"]]
Example: [[EMBED_TEMPLATE_VECTOR="{"template":"Test_GSK_MARKET_ADVANTAGE_Products.txt","prompt":"This is a test sentence","resultcount":5,"similarity_threshold":0.1, "fallback":"vector_Products.txt"}"]]
EMBED_TEMPLATE_VECTOR_NOFAIL
This runs and passes the test successfully, even if there is no template or a fallback one, as mentioned.
Syntax: 
[[EMBED_TEMPLATE_VECTOR_NOFAIL="{"template":"template_name","prompt":"user_prompt","resultcount":{{Count}},"similarity_threshold":{{Thershold_value}}, "fallback":"fallback_template_name"}"]]
Example:
[[EMBED_TEMPLATE_VECTOR_NOFAIL="{"template":"Test_GSK_MARKET_ADVANTAGE_Products.txt","prompt":"This is a test","resultcount":5,"similarity_threshold":0.1, "fallback":"vector_Products.txt"}"]]
Note: Directives are case-sensitive and should be used exactly as defined.

EMBED_NAMESPACE_VECTOR
This is similar to an Embed template vector, but this takes the vectorized value of all the templates under a particular Namespace for processing.
You can provide either the namespace name or the namespace UUID using the same key, namespace.

To improve relevance and retrieval accuracy, we introduced a new boolean flag:
is_manifest_search = True / False
This flag enables the discovery and ranking of semantic corpora using manifest templates.
When this flag is set to False (default), embedding occurs as it normally does.
When this flag is set to True, the directive analyzes the input prompt using manifest templates and dynamically determines the most relevant corpora instead of relying on a single predefined corpus.
When this flag is enabled, a predefined template named SearchManifestByUserQuery is executed every time. This template evaluates all related manifest templates and returns an array of selected manifests, which are then used for the actual search.
Example: "content": "\n[\n {\n \"corpora_name\": \"Fashion_Search\",\n \"namespace_name\": \"search_manifest\",\n \"namespace_uuid\": \"ns_f37c37ee-8379-4f22-85b2-0bee408cb9c6\"\n }\n]\n",
Only the manifests in this array will be used to retrieve data for the source template in which the embed directive is used.
Reranking:
Reranking is the process of re-evaluating and reordering a set of results that an initial search or retrieval model previously retrieved. The re-rank key can be set to true to activate this feature; it is set to false by default. This option is available for both EMBED_TEMPLATE_VECTOR and EMBED_NAMESPACE_VECTOR.
Sample:
[[EMBED_NAMESPACE_VECTOR="{"namespace":"{{c}}","prompt":"{{sentence}}","resultcount":100,"similarity_threshold":0.3,"re_rank":true}"]]
'k' parameter:
The k parameter (often written as top-k) defines the number of candidates that are considered or retained at a given step. The k key can be configured to activate this parameter, and it is available for both EMBED_TEMPLATE_VECTOR and EMBED_NAMESPACE_VECTOR. This option is accessible only when the re-rank feature is enabled.
Sample:
[[EMBED_NAMESPACE_VECTOR="{"namespace":"{{c}}","prompt":"{{sentence}}","resultcount":100,"similarity_threshold":0.3,"re_rank":true,”k”:25}"]]

EMBED_TEMPLATE_CHAIN (PARALLEL EXECUTION)
This is a directive where we will pass a sequence of templates that conducts a step-by-step process for constructing a JSON response using a combination of static and dynamic variable assignments.
Syntax:
[[EMBED_TEMPLATE_CHAIN=”[“template1”],[“template2”],[“template3”,”template4”,”template5”],[“template6”,”template7”],[“template8”]”]]
Sample:
[[EMBED_TEMPLATE_CHAIN="["chain fast_intent"],["chain entity_product","chain entity_geography"],["fast_pos_measure_identifier.txt"]"]]
Each array (group) is executed sequentially, meaning the next array will start only after the previous array has completed.
The bootstrap template (the template containing the EMBED Directive) initializes the execution.
Its parameters are passed to the first array of templates.
After each array completes:
All variables returned from those templates are merged with the existing parameter set.
The combined parameters are passed to the next array.
If multiple templates exist in the next array, the full accumulated parameter set is passed to all of them.
Within each array, the templates are executed in parallel, meaning all templates in that group run simultaneously.

Tool functions:
Implemented end-to-end support for tool execution within the chatbot flow, ensuring that tool results are passed back to the LLM and the final response is streamed to the user.
•	User asks a question.
•	LLM detects the need for a tool invocation based on the user query.
•	The system executes the required tool (external function).
•	Tool results are passed back to the LLM, allowing the model to continue reasoning.
•	The LLM generates the final polished answer, incorporating the tool output.
•	The entire response is streamed, ensuring a smooth and real-time chatbot experience.
To create one, navigate to the Tool function menu on the left pane and click on the ‘Create’ button.
 
Tool Name: Name of the tool
Tool URL (POST): URL of the function call (should be in POST method)
Sample: https://advantageqa2.iriworldwide.com/alerts-dev/services/alerts/fetchJSONLinesForQuery
Description: Provide the description related to product information, sales information, or top brands, to execute this function.
Parameter Key: Name of the Parameter
Parameter Type: Either it could be a string or a number
Description: Based on this description, the param keys' respective value from the prompt or user input will be extracted.
Required: This field specifies which parameters must always be included when calling the function.
Additional Properties: 
In Tools (function definitions), additionalProperties is used to indicate whether the function can accept extra fields beyond those explicitly defined in the schema.
Purpose of additionalProperties in Tool Functions
When defining a tool/function schema:
•	additionalProperties: false
→ The model is not allowed to send any extra parameters other than what you explicitly defined.
→ This is strict validation and helps avoid mistakes or unexpected inputs.
•	additionalProperties: true
→ The model can send extra fields that are not defined in the schema.
→ Useful when your function may need dynamic or optional parameters not known beforehand.
Best Practices:
•	There is no token limit for the description, but it is recommended to keep it short.
•	The description should be 2–3 lines, explaining what the function does so the model knows when to call it.
•	We should not provide complex prompts; if we do, the model may become confused about when to use the tools.
•	We can also include example instructions that explain the purpose of the function.

MCP SERVER
It is a stateless MCP server that allows an LLM to securely execute external domain APIs through dynamically configured tools. It acts as a controlled bridge between the LLM and external APIs.
 
In our current system. We have two types of MCP Servers available.
1.	LLM_MESH_MCP_SERVER (which is our own custom MCP server)
2.	Public MCP Server.
LLM_MESH_MCP_SERVER
This LLM_MESH_MCP_SERVER is non-editable. Here, the user can select only the allowed tools parameter that we configured on the Tools function page.
 
Parameters
•	Tool name: MCP Server name
•	Tools URL: Public URL of the MCP server
•	Description: Description tells what the MCP Server will do.
•	Required Approval: Define user permissions before sharing any data with the MCP Server
•	Allowed tools: We can select one or more tools which we will invoke from the function tools, get parameters of those tools, and map them tothe  execute_function_tool tool present inside the MCP server. Basically, LLM will be forced to use the selected tools
Code Working Logic Overview
1. MCP Server Initialization
The system initializes a FastMCP server with a specific server name.
It also creates a reusable asynchronous HTTP client to make external API calls.This MCP server exposes tools that the LLM can invoke.
2. MCP Server Tool
There are two main tools exposed:
•	get_custom_data → Executes a direct API call.
•	execute_function_tool → Dynamically executes one or more tools defined in the allowed tools.
When the LLM triggers a tool:
1.	The request reaches the MCP server.
2.	The payload is normalized.
3.	The system processes the request based on the selected tool.
4. Tool Execution Flow (Dynamic Execution)
When execute_function_tool is called:
Step 1: Load Server Configuration
The system:
•	Connects to the database.
•	Loads the MCP server configuration.
•	Retrieves the list of allowed tools.
•	Uses Redis cache if available to avoid repeated DB queries.
Step 2: Loop Through Allowed Tools
For each allowed tool:
•	Load its configuration (from Redis or DB).
•	Build the API payload.
•	Call the external API.
Step 4: Process API Response
•	If the response is valid JSON → return it.
•	If string formatting flag is enabled → format report data into structured text.
Step 5: Aggregate Results
All tool responses are collected and returned as a single structured response back to the LLM.
5. Direct API Tool Flow
When get_custom_data is used:
1.	Payload is normalized.
2.	External API is called directly.
3.	Response is processed.
4.	Result is returned.
6. Authentication Handling
If a user ID is provided:
•	The system retrieves a token.
•	The token is added as a cookie in the API request.
•	This allows secure user-specific API execution.
The LLM does not manage authentication — the backend handles it.
7. Redis Caching
The system caches:
•	MCP server configuration
•	Individual tool configuration
If data exists in Redis:
•	It is returned immediately.
If not:
•	It is fetched from the database.
•	Stored in Redis for future use.
This reduces latency and improves performance.

Regression Test Suite Creation
This section outlines the test cases designed to validate the functionality, performance, and reliability of Prompt Regression Testing.
Navigate to Regression testing from the Left pane and Hit Create.
 
In the name section, enter the name of the test case. The model would be automatically set to ‘gpt-4-turbo ‘ and we can change it later on as desired using the dropdown.
 
The output type toggle is used to change the type of the result based on the template. It could be either a Text or a Json.
 
The Prompt Source toggle is used to either choose a Template that already exists or create a Manual input.
If the toggle is set to Manual, a Text box will appear where we will enter the Input prompt.
 
If the toggle is set to Template, a dropdown will appear with a list of all templates available.
 
In the ‘Cases’ menu, there are 3 options.
The case Id would be generated by default with an unique identifier. We can later change this to any desired Id.
 
If there were parameters available in the selected template, we would need to pass them in a Json format.
 
For example, in the template Reworder.txt, the variables would be as follows:
------------------
{
"locale": "en-US",
"sentences": "Even though the weather was bad, we decided to go hiking because we had already planned the trip weeks in advance."
}
------------------
 
In this case, the result would be of type Json. So we need to set the Output type toggle to Json and the Expected output should be as follows:
------------------
{
"results": [
{
"original_sentence": "Even though the weather was bad, we decided to go hiking because we had already planned the trip weeks in advance.",
"new_sentence": "Despite the bad weather, we went hiking as planned weeks ago."
}
]
}
------------------
This step is common for both Manual/Template selection. The complete setup would look like this:
  
We can also add multiple cases by using the add button at the end.
 
Once we hit submit, an entry will be created with the model, created at and updated at timestamps. Note: The updated at timestamp would be using the created at time on the first time of creation.
 
When we hover on to a test case, the hover prompts will include, ‘Edit’, ‘Run’ and ‘Run History’ buttons./svc-gettemplate-by-namespace: This retrieves all the templa
 
We can edit the test case, make any necessary changes, and saveit. Once we hit the Run button, the ‘Run Test Suite’ page will be opened where all the test cases will be listed with the respective id’s and Run button will be available on the Top.
 
Initially the status would be Idle.
Once we hit the run button, the status will change to ‘In Progress’ and the system will start to execute the template with provided variables and compares the result with Deterministic comparison technique and Semantic similarity analysis.
After completion, the status will change to Success/Failure based on the Execution.
If the result is Success, there will be a ‘Show comparison’ button appearing before the Run button as shown below.
 
When we click on the ‘Show Comparison’ button and then click on the accordion on the desired Id, the ‘Result Diff’ would appear with the similarity score and a table which has the Expected results and Outcome.
 
We we hit the back button, the system navigates to the Run History page, where we could see all the results listed. When you hover on any of those, we will have the show results button to take you the respective page.

Notes:
•	Model Selection:
Use any available model from the following endpoint:
GET /providerModels/models
In the JSON response, use the value of the model_name key as the model input.
•	is_json:
When set to true, the output will be returned in JSON format. If false, the output will be plain text.
•	use_template:
Indicates whether to use a template during execution. Set to true to include a template.
•	prompt:
Used when not using a template. Allows manual input of the prompt.
•	source_template_id:
Refers to the ID of the template selected from the UI. This template will be used for the test suite.
•	name:
Specifies the name of the test suite being created.
•	tolerance_level:
A threshold used to determine pass/fail status for a test case. If the result value is below this threshold, the case is marked as failed.
•	cases:
Each case includes:
o	expected_output: The expected result.
o	input_parameter: The input to be tested.
o	id: A unique identifier for the case. This ID must be unique across all existing and new cases.
•	is_concurrent:
Determines whether the test cases should be executed in parallel.
o	When set to true, all cases will run concurrently.
o	When set to false, cases will execute sequentially, one after another.

Execution Endpoint - API SECRETS
To create an API secret key, click on the API Secrets menu in the left pane, then click Create.
 
 
 
Enter a name for the key and set an expiry date. This can be modified at any time.
 
The value of the key will be auto-assigned and can be copied using the copy button next to it.
 
 
 
You can also set a key to inactive status by using the Change Status button in the hover menu for each key.
 
Once you have obtained the key, add an Authorization header using the following format: ‘Bearer’ followed by a space and then the key.
Example: Bearer aashLBljlBLJBljbLJBLJblbLJBHGVkhvks
 
Upload the raw JSON as the request body and hit the execution endpoint from Postman.
 
 
A response will be returned only if a valid auth token is provided. If an invalid auth token is used, the system will return an 'Unauthorized' error.
